# Digital Claims Assistant Рђћ ОбОЋОќОе ОћОфОЉОЎОбОЋОф ОћОЊОЎОњОЎОўОюОЎ

AI claims-collection + management system for **Israeli insurance agents**, built with Claude.

> This file is the always-loaded project brief. The `docs/` folder is the **live source of truth** Рђћ several docs are updated more often than this brief, so when a detail here disagrees with a doc, trust the doc.

## What it is
A client who had a car accident messages their agent ("ОбОЉОеОфОЎ ОфОљОЋОаОћ"). The system runs the full info-collection flow (injuries, car photos, driver's license, vehicle registration, 3rd-party details, event summary), classifies the claim, auto-fills the **"ОћОЋОЊОбОћ ОбОю ОфОљОЋОаОћ"** (accident-notice) form, and hands the agent an organized, ready-to-handle case file + a status dashboard. Replaces hours of WhatsApp/phone chasing. It is a claim *management* system, not just collection.

## Stage
Validation-first, with a warm design partner (a known insurance agent). Goal: prove agents will **pay** before building heavily. Current status lives in `docs/validation-guide.md` and `docs/assumptions-canvas.md`.

## Stack (approved)
Next.js + TypeScript + Supabase (Postgres/Auth/Storage) + Anthropic Claude SDK + Vercel. Hebrew/RTL from day one. Handles sensitive PII (ID, license) under ОЌОЋОД ОћОњОаОф ОћОцОеОўОЎОЋОф.

## Domain essentials
A claim has 4 tracks (`claim_type`): `own_policy`, `third_party_report` ("ОЊОЋОЌ ОцОеОўОЎ"), `third_party_settlement` ("ОћОАОЊОе"), and `unknown` (default, revisable). AI proposes the classification, the agent confirms. Third-party is the sharpest differentiation. Accident-notice forms are per-insurer but ~80% shared fields, and are flat PDFs (no AcroForm) Рєњ fill via **text overlay at coordinates**, not field-fill. Details in `docs/claim-management.md` + `docs/form-field-map.md`.

Shared wizardРєћserver mapping lives in `web/src/lib/collection/claim-state.ts` (`State` + `toClaimData`). **Classification + checklist**: a **two-layer classifier** (`web/src/lib/claims/classify.ts`) Рђћ the LLM only emits narrative signals (`incident_kind`, `inferred_fault`); a deterministic pure function owns the track + confidence + report-vs-settlement recommendation + viability/fault-mismatch warnings. A **dynamic per-track checklist** (`web/src/lib/claims/checklist.ts`, `computeChecklist`) sections items (base/late/conditional/milestone), auto-checks doc items from `claim_documents`, drives conditional items off circumstance flags (migration `004`: `theft`/`lien`/`business_use`/`policy_activated`/`garage_network_rider`), and persists milestone ticks in `claims.checklist_state`. Agent confirms the track via `PATCH /api/claims/[id]/classify`, toggles milestones via `PATCH /api/claims/[id]/checklist`. Both surface on `/dashboard/[id]` (`ClaimTypeConfirm`, `ChecklistPanel`, `AgentDocUpload`). **Task engine** (`web/src/lib/tasks/`, migration `006`): a pure, idempotent `advanceTasks` (`engine.ts`) over a declarative per-track rule table (`templates.ts`) turns events (`claim_submitted`/`track_confirmed`/`milestone_ticked`/`doc_uploaded`) into `{ spawn, complete, statusAdvance }`; `runEngine` (`runner.ts`) applies it **best-effort** (never fails the mutation) inline from the submit/classify/checklist/documents routes, advancing claim status forward-only via compare-and-set. Agents add/edit ad-hoc tasks (`source='manual'`, never auto-completed) via `POST /api/claims/[id]/tasks` + `PATCH /api/claims/[id]/tasks/[taskId]`; surfaced by `TasksPanel` on `/dashboard/[id]` and a next-task column on the dashboard list. The dashboard index also carries a **morning brief** ("ОЉОЋОДОе ОўОЋОЉ" Рђћ `web/src/lib/brief/` Рєњ `MorningBrief`, supersedes the old follow-ups digest): an AI-tiered triage of all open claims. Per claim `facts.ts` builds a fact sheet + `score.ts` a deterministic priority score (overdue tasks / blocking docs / staleness / urgent / unclassified); `rank.ts` asks Claude for a per-claim tier (`act_now`/`this_week`/`waiting`/`ok`) + one Hebrew reason + soft flags (`sanitizeSignals` gates the output, `fallbackTier` gives a rules-only tiering when AI is unavailable); `brief.ts` (`getOrCreateBrief`) is best-effort Рђћ cache-or-compute per agent per UTC day (`agent_briefs`, migration `007`; never persists a degraded brief), returns `null` on failure so the dashboard still renders; `POST /api/brief/refresh` recomputes. The wa.me chase copy (`chaseMessage`/`chaseHref`, alongside `waPhone`) lives in `web/src/lib/wa.ts`. Unit-tested with Vitest (`web/vitest.config.ts`). The detail page is a **cockpit**: a hero (identity + status badge + days-open + AI one-liner) over a **readiness strip** (`ReadinessStrip`) that reads the checklist to say whether the claim is submittable Рђћ offering a one-click **WhatsApp doc chase** (pre-filled with the blocking items + the client's `/c/[token]` link) when blocking docs are missing, or an advance-next-milestone button otherwise Рђћ plus an **agent notes** scratchpad (`NotesPanel` Рєњ `POST /api/claims/[id]/notes`, backed by the `claim_notes` table, migration `005`). `FormGenerator` insurer options are server-derived from the `formfill` template registry.
